# 03 - db.js：SQLite 数据库初始化

> 视频讲解稿 · 对应文件：[backend/db.js](../../backend/db.js)

---

## 开场

这一篇我们进入数据持久化层。数据库选型我们用了 SQLite——一个不需要安装、不需要配置、整个数据库就是一个文件的嵌入式关系数据库。配合 `better-sqlite3` 驱动，所有操作都是同步的，代码简洁直观。

---

## 逐行讲解

### 1. 引入依赖

```javascript
const Database = require('better-sqlite3');
const path = require('path');
```

- **`better-sqlite3`** — 这是我们 npm install 的第三方包。返回的是一个构造函数 `Database`。
- **`path`** — Node.js 内置模块，处理文件路径，跨平台兼容（Windows 用 `\`，Linux/Mac 用 `/`，`path.join()` 自动适配）。

---

### 2. 连接数据库

```javascript
const dbPath = path.join(__dirname, 'blog.db');
const db = new Database(dbPath);
```

- **`__dirname`** — 当前文件所在目录的绝对路径，即 `<项目>/backend/`
- **`path.join(__dirname, 'blog.db')`** — 拼接出 `<项目>/backend/blog.db`
- **`new Database(dbPath)`** — 如果文件不存在，SQLite 会自动创建一个新的数据库文件；如果存在，就打开已有文件。

> **关键设计选择：为什么数据库文件放在 backend 目录下？**
> 因为这是后端专属的数据，不随前端一同部署。同时 `__dirname` 保证无论从哪里执行 `node` 命令，路径都是正确的。

---

### 3. PRAGMA 设置

```javascript
// 开启 WAL 模式提升性能
db.pragma('journal_mode = WAL');
db.pragma('foreign_keys = ON');
```

#### WAL 模式（Write-Ahead Logging）

SQLite 默认使用"回滚日志"模式（DELETE mode）。WAL 模式的优势：

| 特性 | 默认模式 | WAL 模式 |
|------|---------|---------|
| 读写并发 | 写时阻塞读 | 读写可同时进行 |
| 写入性能 | 较慢（需要两次文件写入） | 更快 |
| 额外文件 | 只有 .db | .db + .db-wal + .db-shm |

`PRAGMA` 是 SQLite 特有的命令，用于设置数据库级别的配置参数，不是标准 SQL。

#### 外键约束

```javascript
db.pragma('foreign_keys = ON');
```

**SQLite 默认不检查外键约束！** 必须手动开启。如果不开：

```sql
-- 外键约束关闭时，这条语句不会报错！
INSERT INTO posts (title, content, user_id) VALUES ('test', 'test', 999);
-- user_id=999 在 users 表中不存在，但数据照样插入成功
```

开启后，外键引用完整性得到保证：`posts.user_id` 必须在 `users` 表中存在。

### 4. 建表 SQL

```javascript
function initDB() {
  db.exec(`
    CREATE TABLE IF NOT EXISTS users (
      id INTEGER PRIMARY KEY AUTOINCREMENT,
      username TEXT UNIQUE NOT NULL,
      password TEXT NOT NULL,
      created_at DATETIME DEFAULT CURRENT_TIMESTAMP
    );

    CREATE TABLE IF NOT EXISTS posts (
      id INTEGER PRIMARY KEY AUTOINCREMENT,
      title TEXT NOT NULL,
      content TEXT NOT NULL,
      user_id INTEGER NOT NULL,
      created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
      updated_at DATETIME DEFAULT CURRENT_TIMESTAMP,
      FOREIGN KEY (user_id) REFERENCES users(id)
    );
  `);
}
```

#### 逐字段解析

**users 表**：

| 字段 | 类型 | 约束 | 含义 |
|------|------|------|------|
| `id` | `INTEGER` | `PRIMARY KEY AUTOINCREMENT` | 自增主键，每插入一行自动 +1 |
| `username` | `TEXT` | `UNIQUE NOT NULL` | 用户名不可重复、不可为空 |
| `password` | `TEXT` | `NOT NULL` | 存储 bcrypt 加密后的密文 |
| `created_at` | `DATETIME` | `DEFAULT CURRENT_TIMESTAMP` | 默认取当前时间 |

**posts 表**：

| 字段 | 类型 | 约束 | 含义 |
|------|------|------|------|
| `id` | `INTEGER` | `PRIMARY KEY AUTOINCREMENT` | 文章自增主键 |
| `title` | `TEXT` | `NOT NULL` | 标题 |
| `content` | `TEXT` | `NOT NULL` | 正文 |
| `user_id` | `INTEGER` | `NOT NULL` + `FOREIGN KEY` | 关联到 users 表的 id |
| `created_at` | `DATETIME` | `DEFAULT CURRENT_TIMESTAMP` | 发布时间 |
| `updated_at` | `DATETIME` | `DEFAULT CURRENT_TIMESTAMP` | 最后更新时间 |

#### `IF NOT EXISTS` 关键字

```sql
CREATE TABLE IF NOT EXISTS users (...)
```

**这很重要**：第一次运行时建表，之后重启服务器时，因为表已存在，这条语句会被跳过。不会报错，也不会清空已有数据。

#### `AUTOINCREMENT` 关键字

SQLite 中 `INTEGER PRIMARY KEY` 本身就自增，但 `AUTOINCREMENT` 有额外保证：即使你删除了最后一行，新插入的行的 id 也会继续递增（不会重用已删除的 id）。

#### `CURRENT_TIMESTAMP` 默认值

```sql
created_at DATETIME DEFAULT CURRENT_TIMESTAMP
```

插入数据时如果不指定这个字段的值，SQLite 自动填入当前日期时间，格式为 `YYYY-MM-DD HH:MM:SS`。

---

### 5. 执行初始化并导出

```javascript
initDB();

module.exports = db;
```

- **`initDB()`** — 文件被 require 时立即执行，确保表结构就绪
- **`module.exports = db`** — 导出 `Database` 实例，让其他模块（`routes/auth.js`、`routes/posts.js`）可以直接 `require('./db')` 拿到同一个数据库连接

> **单例模式**：Node.js 的 `require` 有缓存机制——同一个文件只会执行一次，后续 `require('./db')` 返回的都是第一次执行后的缓存结果。所以整个应用共享**同一个数据库连接**，不会重复创建连接。

---

## SQLite 数据文件

启动服务器后，`backend/` 目录下会出现：

```
backend/
├── blog.db       ← 数据库主文件
├── blog.db-wal   ← WAL 日志文件
└── blog.db-shm   ← WAL 共享内存索引
```

- `.db-wal` 和 `.db-shm` 是 WAL 模式的自动产物，不需要手动管理
- 要重置数据库，删除这三个文件然后重启服务器即可
- 可以用 [DB Browser for SQLite](https://sqlitebrowser.org/) 等工具直接打开 `.db` 文件查看数据

## 总结

`db.js` 做的事：

1. 确定数据库文件路径 → 连接（或创建）
2. 开启 WAL 模式（性能）和外键约束（数据完整性）
3. 创建两张表（users + posts），`IF NOT EXISTS` 保证幂等
4. 导出连接实例，供路由模块使用
